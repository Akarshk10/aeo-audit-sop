Answer Engine Optimisation (AEO) SOP

Purpose: Standardise how the agency runs AI search / Answer Engine Optimisation (AEO) engagements across every client.

Scope: Applies to all AI search projects from kickoff to monthly reporting.

Owner: AI Search team lead.  Review cycle: Quarterly (protocols in this space change monthly).

Engagement flow:

Part 1 Prompts → Part 2 Audits (Tech / Content / Links) → Part 3 SoW & Actionables → Part 4 Reporting

Part 1: Prompt Shortlisting & Bucketing

Objective: Test the questions real buyers ask AI assistants, not generic keywords. Prompts become the unit of measurement for the whole engagement (audits, gap analysis, and reporting all trace back to this list).

Step 1.1 — Build the buyer profile

  • Sources: client intake call, existing personas, CRM data, sales-call notes, support tickets, win/loss notes.
  • Capture per persona: role, industry, company size, primary problem, buying triggers, objections, decision criteria, alternatives considered.
  • Gate: Send the buyer profile to the client for written approval before generating prompts. No prompt generation starts without this sign-off.

Step 1.2 — Generate buyer-intent prompts across the funnel

Generate prompts for each persona, mapped to six intent buckets. Target 8 to 15 prompts per bucket per persona at first pass, then shortlist.

BucketBuyer stateExample
Problem-AwareKnows the pain, not the solution category"How can I create professional voiceovers for training videos without hiring voice actors?"
Exploration / ResearchComparing solution categories"What are the best AI voice generation platforms for enterprises in 2026?"
ComparisonTwo or more named options"Murf AI vs ElevenLabs: which is better for enterprise teams?"
AlternativeLooking to switch or replace"Best enterprise-ready alternatives to ElevenLabs with strong security and compliance?"
Use-Case DrivenSpecific scenario or job-to-be-done"How can a corporate L&D team produce multilingual training videos at scale?"
Brand / Branded (add)Knows the brand, validating"Is [Brand] good for [use case]? What do reviews say?"

Note: A "Brand / Branded" bucket is added so the engagement can measure how AI assistants describe the client when named directly. This is where sentiment and review signals surface most.

Step 1.3 — Shortlist and bucket

Shortlist criteria, scored 1 to 5:

  • Buyer relevance (does a real buyer ask this?)
  • Commercial intent (proximity to purchase)
  • Brand relevance (can the client credibly answer it?)
  • Competitive exposure (do competitors already win this?)

Keep the top prompts per bucket. Lock the final list in a shared Prompt Tracker (one row per prompt: persona, bucket, score, target answer URL, current AI answer status, competitors cited).

Deliverable: Approved buyer profile + locked Prompt Tracker. This is the input for Parts 2, 3, and 4.

Three verticals, run in parallel, each producing scored findings that feed the SoW.

Part 2a: Technical Audit and Competitor Benchmarking

Important calibration (state this to every client): Shipping these protocols does not guarantee AI citations or referral traffic. The evidence shows they do two things: (1) reduce fetch and parse cost for agents already crawling the site (roughly a 5x payload drop via Markdown negotiation), and (2) make the site legible to systems deciding whether to cite it. Google's own AI optimisation guidance states new machine-readable files are not required to appear in generative AI search. Sell preparation and legibility, not certainty.

Layer model used in this audit

  • Layer 1, the page: what an agent gets when it fetches a URL (semantic HTML, accessibility, stable layout).
  • Layer 2, around the page: what is discoverable without rendering (robots.txt, sitemap, Link headers, llms.txt, .well-known files, Markdown negotiation, cards).

The 9 components

For each: Technical = what it is. Plain (15-yr-old) = the simple version. Audit = how to check it.

1. Semantic HTML

Technical: Use correct elements (<button>, <a>, <nav>, <main>, <label for>). Agents walking the accessibility tree read element roles; a <div onclick> is invisible to them. Add role and tabindex to interactive divs, cursor:pointer on clickables, tap targets 24x24px or larger, no ghost overlays.

Plain: Build the page out of the right Lego blocks so a robot reading it knows a button is a button and a link is a link. A button drawn as a plain box confuses the robot.

Audit: Open Chrome DevTools, view the accessibility tree, confirm nav, main, buttons, and form labels appear with correct roles. Run Lighthouse accessibility. Flag any interactive element missing a role or label.

2. robots.txt with AI bot rules

Technical: Add explicit Allow / Disallow rules per AI user-agent (GPTBot, ClaudeBot, PerplexityBot, CCBot, anthropic-ai, OAI-SearchBot, Google-Extended). Pair the polite robots signal with CDN/WAF enforcement (Cloudflare, AWS, Fastly) for bots that ignore robots.txt.

Plain: A sign on the door listing which AI robots are welcome. Most polite robots read the sign first. A bouncer at the CDN handles the rude ones.

Audit: Fetch /robots.txt. Confirm each major AI agent is named and the rule is intentional. Blocking everything is the loudest way to disappear from AI search, so confirm blocks are deliberate. Check CDN bot settings.

3. sitemap.xml and Link headers

Technical: Maintain a sitemap. Add HTTP Link headers so agents that scan headers find resources without parsing HTML. Use only the four IANA-registered rels that matter for agents: api-catalog (RFC 9727), service-desc (RFC 8631, points to OpenAPI), service-doc (RFC 8631), describedby (RFC 8288, the correct rel for llms.txt). Ignore invented rels like rel="llms" or rel="ai"; scanners ignore them.

Plain: The sitemap is the table of contents. Link headers are sticky notes on the front cover telling the robot where the important stuff is before it reads a single page.

Audit: Fetch the sitemap, confirm it is current. curl -I the homepage, confirm the four valid Link rels are present with correct type= values. Flag any non-registered rels.

4. llms.txt

Technical: A plain-text/markdown file at the domain root summarising what the site is and which pages matter (spec at llmstxt.org). Realistic use today is narrow: coding agents with the docs URL hardcoded in their system prompt fetch it (for example Claude Code). Chatbot grounding at query time is not observably reading it yet. Cost to ship is near zero.

Plain: A cheat-sheet at the front door telling AI what the site is about and which pages are the good ones. Mostly useful right now for coding assistants, but cheap insurance for later.

Audit: Fetch /llms.txt. Confirm it exists, loads cleanly, summarises the site, and lists priority URLs. Run the Lighthouse Agentic Browsing audit (resolves Not Applicable if missing, no penalty).

5. Open Knowledge Format (OKF)

Technical: A bundle of markdown files (one per page), tagged and cross-linked into a graph an agent can walk. Google introduced it June 2026 (v0.1) under Knowledge Catalog. Nothing crawls the web for these bundles yet. Immediate payoff is a clean machine-legible copy of content plus a structural map of internal links, which doubles as a free site-structure audit.

Plain: A neat zip of the whole site rewritten in robot-friendly text, with a map of how pages connect. Nothing fetches it automatically yet, but making it forces you to see your own site structure clearly.

Audit: Check for an /okf/ bundle. If absent, treat as a low-priority early bet. Use a free OKF generator (point it at the sitemap, up to 100 pages) to produce a bundle and use the output as a structural audit.

6. Markdown negotiation

Technical: When an agent sends Accept: text/markdown, return a markdown version instead of HTML. Agents parse markdown far more efficiently (logged drops from ~85KB HTML to ~16KB markdown on the same page, roughly 5x). Cloudflare's "Markdown for Agents" does this automatically; otherwise implement via Worker route, middleware, or framework hook.

Plain: When a robot says "give me the simple text version," hand it the stripped-down page instead of the heavy designed one. It reads faster and cheaper. Highest-value item for content sites.

Audit: curl -H "Accept: text/markdown" <url> and confirm markdown is returned, not HTML. If on Cloudflare, confirm the feature is enabled.

7. MCP Server Card

Technical: If the site exposes an MCP server, agents discover it via /.well-known/mcp/server-card.json (standardising path, SEP-1649) containing serverInfo, transport, capabilities, and tools. Older flat path /.well-known/mcp-server.json is still checked by some scanners; serve both during transition. Only relevant for sites that host an MCP server.

Plain: If the site offers tools a robot can use (search, lookups), this is the menu card telling the robot what tools exist and how to call them. Skip it if the site has no such tools.

Audit: Fetch both well-known paths. Validate JSON shape. If the client has no MCP server, mark Not Applicable, do not flag as a gap.

8. A2A Agent Card

Technical: If the site is itself an agent (offers services other agents call), Google's A2A protocol uses /.well-known/agent.json for discovery (name, description, endpoint, capabilities, version). Bleeding edge, spec still moving; pin to a dated version if shipping. Most content sites do not need this.

Plain: Only matters if your website is itself a robot that other robots can hire. A business card for that robot. Most sites can ignore it.

Audit: Fetch /.well-known/agent.json. Applicable only if the client runs an agent or agent-facing API; otherwise Not Applicable.

9. Content Signals in robots.txt

Technical: A robots.txt extension (IETF draft, contentsignals.org) declaring separate permissions: ai-train=yes|no (training), search=yes|no (citation in AI search), ai-input=yes|no (runtime RAG/grounding). Added as Content-Signal: under a user-agent group. Separates "you can read me" from "you can train on me." On Cloudflare, keep the "Managed robots.txt" toggle OFF if managing the file manually, or it overwrites custom directives.

Plain: Three switches telling AI: you may read me, you may quote me, you may train on me, set independently. Most publishers today allow citation but block training. The client picks their stance.

Audit: Fetch /robots.txt, check for Content-Signal: directives. Confirm values match the client's stated stance. Confirm Cloudflare Managed robots.txt toggle is not overwriting them.

Competitor benchmarking (technical)

  • Run the same scanner pass (isitagentready.com, manual curl) against the top 3 to 5 competitors named in the Prompt Tracker.
  • Record per competitor: which of the 9 components are present, scanner score, Markdown negotiation present yes/no, llms.txt present yes/no.
  • Output: a competitor matrix (rows = the 9 components, columns = client + each competitor) showing where the client is behind. This directly feeds the SoW priority.

Part 2b: Content and Entity Audit

Two workstreams: build topical authority, and close content gaps surfaced by the tracked prompts.

Workstream 1: Topical authority and SEO fundamentals

  • Apply standard SEO content quality across the site: clear topic hierarchy, internal linking, unique demonstrable expertise (first-hand experience, data, named authors), accurate and current information, and entity clarity (consistent brand/product/person naming, About pages, schema for entities).
  • Density rule (evidence-based): AI search filters favour dense pages. Roughly 32% of a page typically survives the AI search filter, and 800-word dense pages reach 50%+ coverage versus 4,000-word pages at 13%. Write tight, front-load the answer, avoid padding.
  • Map the site's existing content to topics and identify the topical clusters where the client can credibly claim authority.

Workstream 2: Content gap analysis from tracked prompts

For every prompt in the Prompt Tracker, classify the content gap into one of three states:

Gap stateDefinitionAction
1. No coverageBuyers ask it via AI, no page on the site answers itCreate new content targeting the prompt
2. Shallow coverageTopic exists but lacks depth to compete for the keyword or appear in LLM resultsExpand, add depth, data, examples, structure
3. Zero-search / persona-ledPrompt has little or no traditional search volume but matches a real buyer personaCreate persona-driven content; do not gate on search volume

State 3 note: AI assistants surface answers to questions that never appear in keyword tools. Build content for the persona's real questions even at zero measured volume.

Workstream 3: Query fan-out coverage

  • For each core topic, map the query fan-out (the related sub-questions an AI expands a query into) using a tool such as queryfan.com.
  • Check whether the site has content covering each fanned-out sub-query.
  • Output: a fan-out coverage map per core topic, flagging uncovered branches as content gaps feeding the SoW.
Deliverable: Topical authority map + prioritised content gap list (each item tagged state 1/2/3) + query fan-out coverage map.

Part 2c: Links, Brand Mentions, Reviews, and UGC Sentiment

Objective: AI assistants weight third-party signals (mentions, reviews, UGC) heavily when deciding what to cite about a brand. Audit where the brand is present, absent, under-ranked, and how it is talked about.

2c.1 Brand mention gap analysis

  • Competitor-present, client-absent: find content, listicles, and roundups where competitors are mentioned but the client is not. These are mention-acquisition targets.
  • Client-present-but-weak: find content where the client is mentioned but ranked or positioned below the top 3. These are position-improvement targets (outreach, content updates, getting upgraded in existing listicles).
  • Output: two lists, "missing mentions" and "weak mentions," each with source URL, competitors present, and outreach angle.

2c.2 Ratings and reviews

  • Audit presence and score on platforms AI assistants read: Trustpilot, G2, TripAdvisor, and any vertical-relevant equivalents.
  • Record per platform: profile exists yes/no, rating, review count, recency of reviews, and comparison to top 3 competitors.
  • Flag platforms where the client is absent or materially below competitors.

2c.3 UGC presence and sentiment

  • Audit brand presence on UGC platforms AI assistants pull from: Reddit and Quora primarily.
  • Use the in-house tracking tool to record where the client and competitors appear, and run sentiment analysis on the mentions.
  • Output: UGC presence + sentiment comparison (client vs competitors), flagging negative-sentiment threads (response/remediation targets) and absent high-relevance threads (participation targets).
Deliverable: Mention gap lists, review-platform scorecard, and UGC presence + sentiment comparison.

Suggested additions to consider for the SoW

  • Quarterly re-audit cadence, since protocols change monthly.
  • CDN/WAF bot enforcement setup as a paired item with robots.txt.
  • Schema/entity infrastructure work (indirect value as entity infrastructure feeding Knowledge Graph, even though direct citation evidence is weak).
  • A "prompt expansion" line item to add newly discovered buyer prompts each quarter.
Prioritisation rule: Rank by (commercial intent of the source prompt) × (competitive exposure) × (effort inverse). Ship the minimum-viable technical set first, then high-intent content gaps, then mention/review/UGC work.

Reporting notes

  • Lead with prompt visibility and Share of Voice; these are the outcomes clients care about.
  • Always show client vs top 3 competitors side by side.
  • State clearly each cycle: crawl rate and surfacing/citation rate are different metrics. High crawl volume does not equal citations.
  • Do not promise specific citation outcomes in any report. Report what moved and why.

Appendix: Standardisation Checklist (per engagement)

  1. Buyer profile approved by client.
  2. Prompt Tracker locked with 6 buckets, scored and prioritised.
  3. Technical audit run (3 scanners + manual curl) for client and top 3 competitors; 9-component matrix complete.
  4. Content audit complete: topical authority map, gap list (states 1/2/3), fan-out coverage map.
  5. Links/brand audit complete: mention gaps, review scorecard, UGC sentiment comparison.
  6. SoW built with every line traced to a finding and prioritised.
  7. Reporting dashboard configured with core + supporting metrics.
  8. Re-audit scheduled (quarterly).
